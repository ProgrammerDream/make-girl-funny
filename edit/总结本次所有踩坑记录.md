# Playwright + CDP screencast + ffmpeg 自动剪辑 4K 竖屏视频踩坑记录

> 项目从零到出片踩了 24 个坑，覆盖 ffmpeg 参数、Playwright 录屏、CDP screencast、4K 适配、鼠标拟人化、TTS 配音和音视频拼接等。
> 每个坑都给出"看似合理但其实错"的中间方案，避免后人重复踩。

---

## A. ffmpeg / 编码相关

### 1. `force_original_aspect_ratio=cover` 不存在

**现象**：`Error applying option 'force_original_aspect_ratio': Invalid argument`

**根因**：`cover` 是 CSS 值，ffmpeg scale 滤镜只认 `increase` / `decrease` / `disable`。

**正解**：`scale=W:H:force_original_aspect_ratio=increase,crop=W:H` 等效 cover。

---

### 2. CRF=20 给 4K → 码率太低，看着和 1080p 一样

**现象**：final.mp4 是 2160×3840 但只有 29MB / 1.85Mbps。

**根因**：CRF 是"视觉质量恒定"，但同一 CRF 在 4K 分配的比特率远低于真实需要。4K 像素是 1080p 的 4 倍，CRF=20 没给够 4 倍预算。

**正解**：4K 时 CRF 从 20 降到 16。参考：YouTube 4K 推荐 35~45Mbps，抖音 4K 上传约 20Mbps。

---

### 3. 75MB 4K 文件以为是质量问题

**现象**：4K 视频只有 75MB / 115s，体感太小。

**真相**：x264 对屏幕录制的帧间压缩极其高效。
- 大量静止画面，每帧只编码极少差分
- 4K + CRF 16 的屏幕内容平均码率 3~10 Mbps 是正常的
- 真实 4K 实拍 30~80 Mbps，但屏幕录制不能拿这个对比

**结论**：文件小不代表质量差。但定位时意外发现 raw 实际是 1080p（引出坑 13、14）。

---

## B. Playwright 录屏方案选择

### 4. Playwright `recordVideo` 不给真 4K，且 fps 受限

**现象**：录出来 webm 分辨率永远是 viewport CSS 像素（1080×1920），fps ≈ 25 不可调。

**根因**：Playwright 录屏走内部 VP8/VP9 通道，和 CDP screencast 不是同一管道，不支持物理像素输出。

**正解**：**弃用 `recordVideo`**，改 CDP `Page.startScreencast` + ffmpeg stdin 编码。

---

### 5. scale=2 + dsf=2 + recordVideo → 内容只占画布 1/4

**现象**：录屏画布 2160×3840，但页面内容只在左上角 1080×1920。

**根因**：recordVideo 的 size 是物理画布，但 viewport 决定内容布局范围。dsf 只影响像素密度，不影响 recordVideo 的画布填充。

**结论**：此路不通，配合坑 4 切换到 CDP screencast。

---

### 6. viewport 直接开 2160×3840 → 页面字体变小一半

**现象**：页面以为屏幕真有 2160px 宽，所有元素按比例缩小。

**根因**：响应式页面的 `max-width: 800px` 在 2160 视口里占比变小，rem/em 不变但相对画面比例小了。

**正解**：viewport 永远保持 1080×1920（设计尺寸），4K 通过 dsf + screencast 物理像素获得。

---

### 7. 用 `Page.captureScreenshot` 主动拉帧 → 4K 下渲染进程崩溃

**思路**：被动 screencast 静态页不推帧（见坑 9），改 `setInterval(33ms)` 主动 captureScreenshot。

**结果**：30 秒内崩溃：
```
actions error: page.waitForTimeout: Target page, context or browser has been closed
```

**根因**：4K JPEG 单帧 ~1MB，30fps = 30MB/s 走 devtools IPC。captureScreenshot 触发完整 paint pipeline（screencast 是合成器零拷贝），渲染进程被压崩。

**教训**：4K 下 captureScreenshot 不能高频调用。screencast 是唯一可行路径。

---

## C. CDP screencast 隐含行为

### 8. screencast 不传 `maxWidth/maxHeight` → 输出只有 CSS 像素

**现象**：dsf=2 但 screencast 输出仍是 1080×1920。

**根因**：`Page.startScreencast` 默认按 viewport CSS 像素截图。必须显式传 `maxWidth` / `maxHeight` 为物理像素尺寸才会按 dsf 输出。

**正解**：
```js
await client.send('Page.startScreencast', {
  format: 'jpeg',
  quality: 85,
  maxWidth: 2160,    // = viewport.width * scale
  maxHeight: 3840,   // = viewport.height * scale
  everyNthFrame: 1
});
```

> 但只传这个还不够，见坑 14。

---

### 9. CDP screencast 静态页推不出帧

**现象**：`hook.html` / `awards.html` 录出 0 帧（空文件）。

**根因**：screencast 基于 **dirty rect** 触发——浏览器合成器认为没"脏区域"就不推帧。纯静态 HTML（即使有 CSS 动画）有时不算"脏"。ffmpeg 一直收不到 jpeg，进程退出。

**错误尝试**：调 `quality` / `everyNthFrame`。无效。

**正解**：注入心跳元素，每帧微动让浏览器认它为"脏"：
```js
const heart = document.createElement('div');
heart.style.cssText = 'position:fixed;left:0;top:0;width:1px;height:1px;opacity:0.01;will-change:transform';
document.body.appendChild(heart);
let n = 0;
setInterval(() => {
  heart.style.transform = 'translateX(' + ((n++) % 2) + 'px)';
}, 33);
```

---

### 10. rAF 心跳 + ffmpeg 30fps → 时长翻倍

**现象**：5s 的 hook 录出 9.6s，13s 的 seg-10 变成 25.8s。

**根因**：
- 心跳用 `requestAnimationFrame` → headless ~60fps
- screencast 跟着每个合成帧推 → 60fps jpeg 流
- ffmpeg `image2pipe` 不认 wallclock 时间戳，按 1/30s 编码
- 60fps 输入按 30fps 写出 → 时长 ×2

**正解**：心跳改 `setInterval(33ms)`，让 screencast 推帧节奏 ≈ 30fps。

> 曾试 `-use_wallclock_as_timestamps 1 -vsync cfr` 让 ffmpeg 用墙钟，image2pipe + mjpeg 路径下几乎无效，必须从源头控制推帧节奏。

---

### 11. headless rAF 在某些版本被节流

**没踩进去但要警觉**：headless chromium 没可见动画时 rAF 可能被节流到 1Hz。

**正解**：心跳元素加 `will-change: transform` + `setInterval` 替代 rAF，同时规避坑 10 和这个风险。

## D. 浏览器 4K 渲染配置

### 12. headless chromium 的 `deviceScaleFactor` 不影响 screencast

**现象**：`newContext({ deviceScaleFactor: 2 })` 但 screencast 输出还是 1080×1920。

**根因**：headless chromium 长期 bug——`Emulation.setDeviceMetricsOverride`（playwright `deviceScaleFactor` 底层走这条）只影响 `window.devicePixelRatio` 和 DOM 行为，**不影响合成器输出尺寸**。

**正解**：必须用 Chromium 启动参数：
```js
chromium.launch({
  headless: true,
  args: ['--force-device-scale-factor=2']
})
```
这个标志让整个浏览器进程在 2x dsf 下渲染，合成器输出物理像素图，screencast 直出 4K。

**教训**：playwright 的 `deviceScaleFactor` API 和 Chrome 启动参数不是一回事。

## E. 时长 / actions 控制

### 13. `Promise.race([actions, duration])` 切断 actions

**现象**：很多按钮没点到，日志一堆 `actions error: Target page closed`。

**根因**：原代码到 duration 秒就关浏览器：
```js
await Promise.race([actionsPromise, setTimeout(duration*1000)]);
stopped = true;
await browser.close();
```
actions 实际耗时 ≈ duration（很容易超），最后几个 click 和反馈帧被吞。

**正解**：actions 必须跑完，duration 改"保底最短"语义：
```js
await Promise.race([
  actionsPromise,
  new Promise(r => setTimeout(r, hardCapMs))  // 2.5x duration 兜底防卡死
]);
const elapsed = Date.now() - startMs;
if (elapsed < minDurationMs) {
  await new Promise(r => setTimeout(r, minDurationMs - elapsed));
}
```

## F. 鼠标 / 交互真实感

### 14. 硬编码坐标点不到按钮

**现象**：鼠标在空白处乱点，按钮一个没碰到。

**根因**：每个 html 布局不同，`max-width` 居中后元素实际位置和猜的偏差很大。

**正解**：CSS 选择器 + `getBoundingClientRect()` 取元素中心，加 `clickSel` / `hoverSel` / `chase` 三种动作类型。

---

### 15. "愿意"按钮被"不愿意"挡住点不到

**现象**：DeepSeek/GLM 的逃跑按钮跑到 body 层后 z-index 高，覆盖了"愿意"按钮。`clickSel` 靠坐标点击被拦截。

**正解**：加 `clickJs` 动作——直接 `el.click()` 派发事件，绕过视觉遮挡。

---

### 16. `mouse.wheel` 在 `overflow:hidden` 上不生效

**现象**：`scroll(y)` 没效果，下面的按钮始终不可见。

**根因**：body 是 `overflow:hidden`，wheel 事件不产生滚动。

**正解**：加 `scrollSel` 动作——`el.scrollIntoView({ block: 'center' })` 强制滚到元素。

---

### 17. `smoothMove` 线性插值 → 鼠标像激光笔

**现象**：鼠标移动机械、生硬，一眼就是脚本。

**根因**：原代码 25 步等距线性插值 + 16ms 等步长。真人鼠标有：
- 缓动（起步慢、中间快、终点慢）
- 弧线轨迹（不是直线）
- 微抖动（手不稳）
- 偶尔停顿（看清楚再走）

**正解**：
```js
const eased = t < 0.5 ? 4*t*t*t : 1 - Math.pow(-2*t+2, 3)/2;     // ease-in-out cubic
const off = Math.sin(Math.PI * t) * arc;                          // 贝塞尔弧线
const jitterX = (Math.random() - 0.5) * 1.2;                      // 手抖
if (Math.random() < 0.06) await page.waitForTimeout(60+Math.random()*80);  // 停顿
```
步数按距离自适应，避免短距离也走 25 步。

---

### 18. script.json 漏点按钮

**现象**：很多页面只点了一半交互。

**根因**：写脚本时凭印象写，没逐个 grep HTML 里的 `onclick`/`id="*Btn"`。

**正解**：每页 grep 一遍所有交互元素：
```bash
grep -nE 'onclick|id="[a-zA-Z][^"]*Btn"' xxx.html
```
然后照 selector 列表逐个补 click action。每个 click 前加 `scrollSel`，让目标进视口再点（getBoundingClientRect 在屏外会返回负坐标，click 会落空）。

## G. 其他

### 19. emoji 显示成方块/问号

**现象**：emoji 显示为 `□` 或 `?`。

**根因**：`@import url('fonts.googleapis.com/...')` 引入的 Comic Neue 只支持拉丁字符，国内访问 Google Fonts 超时后 fallback 到不支持 emoji 的字体。

**正解**：删掉 Google Fonts import，font-family 加 `Segoe UI Emoji, Apple Color Emoji, Noto Color Emoji`。

## H. TTS 配音 / concat 时长相关

### 20. `-c:v copy + -t` 不是真截视频

**现象**：每段单独看时长像是对的，但 concat 后 final 被拉长，后半段出现静止画面。

**根因**：对 H.264 用 `-c:v copy -t` 时，ffmpeg 可能只改容器层 duration，底层 packet 并没有真的被裁掉。concat demuxer 按 packet 时间戳拼接，原视频尾巴会回来。

**正解**：视频长于配音时，不截视频，直接保留完整视频 packet；音频用 `apad + atrim` 补齐并截到视频时长。配音长于视频时，走 `tpad` freeze 并重编码视频。

---

### 21. 只看 stream duration 会误判，必须看 packet

**现象**：`ffprobe stream=duration` 显示 voiced 段音视频接近一致，但 final 仍然多出几十秒。

**根因**：stream duration 只反映流级别估算，不能暴露 packet 的真实 `pts_time + duration_time`、DTS 是否单调、拼接点是否被改写。

**正解**：排查拼接问题时同时看最后一个 packet：
```bash
ffprobe -v error -select_streams a:0 \
  -show_packets -show_entries packet=pts_time,duration_time \
  -of csv=p=0 out/final.mp4 | tail
```

---

### 22. `apad + -shortest` 不能当精确截断

**现象**：音频补静音后，看似有 `-shortest`，但 concat 后仍可能出现音频尾部异常。

**根因**：`apad` 是无限音频源，`-shortest` 是输出层停止条件，不等于把滤镜链中的音频流精确裁成目标长度。AAC 编码还会有帧边界 padding。

**正解**：补静音后显式加：
```bash
apad,atrim=0:目标秒数,asetpts=N/SR/TB
```
让音频流在滤镜层被硬截断，再编码。

---

### 23. concat 输入音频格式不一致会把 DTS 改乱

**现象**：voiced 合计约 67s，final 视频流也是 67s，但音频流被拉到 114s，1 分钟后画面静止。

**关键证据**：最终 concat 日志出现：
```text
Reconfiguring filter graph because audio parameters changed to 24000 Hz, stereo, fltp
Non-monotonic DTS; previous: ..., current: ...; changing to ...
```

**根因**：voiced 段是 `24000Hz mono`，黑帧切场 `black.mp4` 曾是 `44100Hz stereo`。concat demuxer 要求各段流参数一致；音频参数中途变化时，ffmpeg 会在拼接点重配滤镜并重写 DTS，导致音轨时间轴被撑长。

**正解**：定义统一音频格式，并让所有配音、静音、黑帧、最终 concat 都保持一致：
```js
const AUDIO_RATE = 24000;
const AUDIO_CHANNELS = 1;
const AUDIO_LAYOUT = 'mono';
```

---

### 24. concat 阶段音频不要 copy，视频可以 copy

**现象**：各段 mp4 单独播放正常，但最终拼接时音频时间轴最容易出问题。

**根因**：`-c copy` 会保留输入段已有的音频 packet 时间戳、编码延迟、padding 和容器 metadata。只要任一段不规整，问题就会累积到最终文件。

**正解**：最终 concat 使用：
```bash
ffmpeg -f concat -safe 0 -i concat.txt \
  -c:v copy \
  -c:a aac -b:a 128k -ar 24000 -ac 1 -shortest \
  final.mp4
```
视频 copy 保持帧不重压；音频重编码生成干净、统一的时间轴。AAC 仍可能多出几十毫秒帧边界 padding，这是正常的，不会变成几十秒。

## 规律总结

| 坑的类型 | 编号 | 本质 |
|---|---|---|
| API 参数认知错误 | 1, 5, 6, 8, 12 | 想当然以为支持某个值/行为，实际文档没写或行为不同 |
| 硬编码 vs 动态查询 | 14, 15, 16, 18 | 猜坐标永远不靠谱，必须运行时读取真实位置 |
| 外部依赖失效 | 19 | 网络字体在离线/国内环境挂掉 |
| CDP screencast 隐含行为 | 4, 8, 9, 10, 11, 12 | Chromium 内部实现的坑，文档几乎不提 |
| 编码 / 容器时间轴 | 2, 3, 20, 21, 22, 23, 24 | CRF、packet、DTS、AAC padding 都不能只看表面 duration |
| 时序 / 生命周期 | 7, 13 | Promise.race / 进程压力下的 race condition |
| 真实感细节 | 17 | 算法对，但参数不像人 |

## 最终方案速查

| 决策 | 选择 | 替代方案为何不行 |
|---|---|---|
| 录屏管道 | `Page.startScreencast` | `recordVideo` 不给 4K 不可调 fps（坑 4） |
| 4K 输出 | `--force-device-scale-factor=2` 启动参数 + screencast `maxWidth/maxHeight` | `newContext.deviceScaleFactor` 不影响 screencast（坑 12） |
| 静态页推帧 | 心跳元素 + setInterval(33ms) | dirty-rect 不触发（坑 9）；rAF 60fps 导致时长翻倍（坑 10） |
| 主动 vs 被动截屏 | screencast 被动 | captureScreenshot 4K 30fps 压崩进程（坑 7） |
| 时长控制 | actions 跑完 + duration 保底 | `Promise.race([actions, duration])` 切断 click（坑 13） |
| 鼠标移动 | 缓动+弧线+抖动+停顿 | 线性插值生硬如激光笔（坑 17） |
| 4K CRF | 16 | 20 在 4K 下码率不够（坑 2） |
| 配音拼接 | 统一 24000Hz mono，最终音频重编码 | 音频格式混用会触发 DTS 改写（坑 23 / 24） |

## 关键 ffprobe 检查命令

每次怀疑录制问题先验证：
```bash
ffprobe -v error -select_streams v:0 \
  -show_entries stream=width,height,nb_frames,duration \
  -of default=nw=1 raw/seg-01.mp4
```
- `width/height` 不是 2160/3840 → 4K 没出来（坑 8 / 12）
- `nb_frames` 远低于 `duration*30` → 静态页问题（坑 9）
- `duration` 远大于 script 设定 → 推帧节奏问题（坑 10）
- `duration` 远小于 script 设定 → race 切断（坑 13）
